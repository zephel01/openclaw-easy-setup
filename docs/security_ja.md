## 概要

openclaw-easy-setup は「セキュリティ・バイ・デフォルト」の方針で設計されています。
ユーザーが意識的に設定を緩めない限り、すべてのセキュリティ機能が有効な状態で起動します。

## 1. ネットワークセキュリティ

### ポートバインド

すべてのポートは `127.0.0.1`（ローカルホスト）にバインドされます。
LAN や外部ネットワークからアクセスすることはできません。

```yaml
# docker-compose.yml
ports:
  - "127.0.0.1:18789:18789"  # OpenClaw Gateway
  - "127.0.0.1:11434:11434"  # Ollama
```

### ファイアウォール設定

追加の防御策として、OS レベルのファイアウォール設定を推奨します。

**Linux (UFW):**
```bash
sudo ufw deny 18789/tcp
sudo ufw deny 11434/tcp
```

**macOS (pf):**
```bash
echo "block in on en0 proto tcp from any to any port 18789" | sudo pfctl -ef -
```

### Docker ネットワーク

コンテナ間通信は専用の Docker ブリッジネットワーク (`openclaw-net`) を使用します。
サブネット `172.28.0.0/16` が割り当てられ、ホストネットワークとは分離されています。

## 2. コンテナセキュリティ

### OpenClaw コンテナ

| 設定 | 値 | 効果 |
|------|---|------|
| `security_opt: no-new-privileges` | `true` | `setuid`/`setgid` による権限昇格を防止 |
| `read_only` | `true` | コンテナのファイルシステムを読み取り専用に |
| `cap_drop` | `ALL` | すべての Linux capabilities を削除 |
| `cap_add` | `NET_BIND_SERVICE` | 低ポート番号へのバインドのみ許可 |
| `tmpfs` | `/tmp:256M`, `/run:64M` | 一時ファイルをメモリ上に制限 |
| `USER` | `openclaw` (非root) | コンテナ内で非特権ユーザーとして実行 |

### Dockerfile セキュリティ

マルチステージビルドを採用し、最終イメージのサイズと攻撃対象面を最小化しています：

- **ビルドステージ**: Node.js + npm で OpenClaw をインストール
- **ランタイムステージ**: 最小限の依存関係のみ（ca-certificates, curl, tini）
- **不要ツールの削除**: apt, dpkg などのパッケージマネージャーを除去
- **tini**: PID 1 として適切なシグナルハンドリングを実現

### リソース制限

```yaml
deploy:
  resources:
    limits:
      memory: 2G      # OpenClaw
      cpus: "2.0"
    # Ollama
    limits:
      memory: 8G
      cpus: "4.0"
```

## 3. OpenClaw アプリケーションセキュリティ

### 認証

```yaml
gateway:
  auth:
    mode: "token"
    token: "<auto-generated-64-char-hex>"
```

- トークンは `openssl rand -hex 32` で生成（256ビットエントロピー）
- `.env` ファイルに保存（権限: 600）
- トークン認証を無効にする場合は明示的に `none` を設定する必要があります

### DM ポリシー

```yaml
messaging:
  dmPolicy: "pairing"
```

| ポリシー | 動作 |
|---------|------|
| `pairing` (デフォルト) | 初回のDMは承認が必要。不明な送信者からのメッセージをゲート |
| `open` | すべてのDMを受け入れ（非推奨） |

### ツール制限

```yaml
tools:
  deny:
    - exec      # シェルコマンド実行
    - browser   # ブラウザ操作
    - cron      # スケジュール実行
```

これらのツールはデフォルトで無効化されています。必要に応じて `config.env` の `OPENCLAW_DENY_TOOLS` を変更してください。

### サンドボックス分離

```yaml
security:
  sandboxInheritEnv: false
```

AIエージェントのサンドボックス環境にはホストの環境変数が渡されません。
これにより、APIキーやシステム情報がエージェント経由で漏洩するリスクを軽減します。

## 4. 認証情報管理

### Docker 環境 (.env)

```
.env                  # 権限: 600 (owner read/write only)
├── ANTHROPIC_API_KEY
├── OPENAI_API_KEY
├── OPENCLAW_AUTH_TOKEN
└── ...
```

- `.gitignore` に登録済み — Git にコミットされません
- `setup.sh` が自動的に `chmod 600` を適用

### ネイティブ環境 (~/.openclaw/)

```
~/.openclaw/           # 権限: 700
├── .env               # 権限: 600 — 基本設定（API キーは secrets/ 推奨）
└── openclaw.json      # 権限: 600 — 設定（トークン含む）
```

### ClawX のキーチェーン統合

ClawX はOSのネイティブキーチェーンにAPIキーを保存します：

| OS | ストレージ |
|----|----------|
| macOS | macOS Keychain |
| Windows | Windows Credential Manager |
| Linux | libsecret |

暗号化方式: AES-256-GCM（テーブルキー + 行単位キー）

## 5. セキュリティチェックリスト

セットアップ後、以下を確認してください：

### 必須項目

- [ ] `.env` ファイルの権限が `600` であること
- [ ] `docker compose ps` でコンテナが正常に動作していること
- [ ] ポート 18789/11434 が外部からアクセスできないこと
- [ ] `./setup.sh --doctor` でエラーがないこと

### 推奨項目

- [ ] OS ファイアウォールでポートをブロック
- [ ] 不要なツール（exec, browser, cron）が deny リストにあること
- [ ] `dmPolicy` が `pairing` に設定されていること
- [ ] `requireMention` が `true` に設定されていること
- [ ] Docker リソース制限が環境に合った値であること

### 定期確認

- [ ] Docker イメージを定期的に更新（セキュリティパッチ適用）
- [ ] API キーのローテーション
- [ ] ログの確認: `docker compose logs openclaw`

## 6. セキュリティに関する設定変更

### 認証を無効にする（開発環境のみ）

```env
OPENCLAW_AUTH_MODE=none
```

> 本番環境では絶対に無効にしないでください。

### ツール制限を緩和する

```env
OPENCLAW_DENY_TOOLS=browser
# exec と cron のみ許可、browser は引き続き拒否
```

### 外部からのアクセスを許可する（非推奨）

`docker-compose.yml` のポートバインドを変更：

```yaml
ports:
  - "0.0.0.0:18789:18789"  # すべてのインターフェースで公開
```

> この設定はリバースプロキシ（nginx/Caddy）+ TLS の背後でのみ使用してください。

## 7. インシデント対応

### APIキーが漏洩した場合

1. 該当プロバイダーのダッシュボードで即座にキーを無効化
2. 新しいキーを設定: `echo "new-key" > secrets/PROVIDER_API_KEY`
3. `docker compose restart openclaw` で再起動
4. Git の履歴を確認し、コミットされていないか確認

### 不正アクセスの疑いがある場合

1. `docker compose down` でサービスを停止
2. ログを確認: `docker compose logs openclaw > incident.log`
3. 認証トークンを再生成: `.env` の `OPENCLAW_AUTH_TOKEN` を更新
4. サービスを再起動
